# PJM Project Explorer — Developer Challenge

## What is PJM?

PJM Interconnection is a regional transmission organization (RTO) that coordinates the movement of wholesale electricity in all or parts of 13 states: Delaware, Illinois, Indiana, Kentucky, Maryland, Michigan, New Jersey, North Carolina, Ohio, Pennsylvania, Tennessee, Virginia, and West Virginia. It operates one of the world's largest competitive wholesale electricity markets.

Renewable energy developers who want to connect a new solar, wind, or storage facility to the power grid must submit an application to PJM's Interconnection Queue. This queue contains thousands of projects at various stages of development — from early-stage applications to completed, operating facilities. Analyzing and tracking these projects is at the core of what REplace does.

This challenge simulates a simplified internal tool for browsing and managing a subset of those projects.

---

## The Challenge

Build a full-stack web application that allows users to browse PJM renewable energy projects, filter them by state or fuel type, view project details, and bookmark projects of interest.

The backend skeleton and data model are provided. The frontend structure is scaffolded. **Your job is to implement the logic.**

## About SQLite and the `database.db` file

SQLite is a lightweight database engine that stores all data in a **single file on disk** — in this case `backend/database.db`. Unlike PostgreSQL or MySQL, SQLite requires no separate server process. The SQLAlchemy library reads from and writes to this file directly, the same way it would talk to a PostgreSQL server in production.

> **Note:** `database.db` is listed in `.gitignore` and must not be committed. It is generated locally when you run the seed script described below.

In a production environment, the same SQLAlchemy code connects to PostgreSQL — only the connection string changes from `sqlite:///./database.db` to `postgresql://user:pass@host/db`.

## Setup & Running

You will need two terminals open simultaneously — one for the backend, one for the frontend.

### Backend

```bash
cd backend
pip install -r requirements.txt

# Create the database and load the 20 sample projects
python seed.py

# Start the API server
uvicorn main:app --reload
```

> **Note:** Depending on your local Python installation, you may need to create and activate a virtual environment before installing dependencies.

```bash
# From the backend directory, create a virtual environment
python3 -m venv .venv

# Activate it on Linux, macOS, or WSL
source .venv/bin/activate

# On Windows PowerShell
.\.venv\Scripts\Activate.ps1

# Install the backend dependencies
python -m pip install -r requirements.txt
```

After activating the virtual environment, continue with the regular backend setup commands.

If the `uvicorn` command is not recognized, run it through Python:

```bash
python -m uvicorn main:app --reload
```

The API will be available at `http://localhost:8000`.
Interactive API docs (auto-generated by FastAPI) are available at `http://localhost:8000/docs` — useful for testing endpoints before connecting the frontend.

### Frontend

Open a second terminal:

```bash
cd frontend
npm install
npm start
```

The app will be available at `http://localhost:3000`.

## Data Model

```
Project:
    id           int      — primary key
    name         str      — project name
    state        str      — state code, e.g. "OH", "KY", "PA"
    fuel_type    str      — "solar", "wind", or "storage"
    capacity_mw  float    — installed capacity in megawatts
    status       str      — "active", "withdrawn", or "completed"
    developer    str      — developer company name
    bookmarked   bool     — default: False
```

## API Endpoints

| Method | Endpoint                       | Description                                                                 |
|--------|--------------------------------|-----------------------------------------------------------------------------|
| GET    | `/health`                      | Health check — already implemented                                          |
| GET    | `/api/projects`                | List all projects. Optional filters: `?state=OH` and/or `?fuel=solar`      |
| GET    | `/api/projects/{id}`           | Get project detail. Returns HTTP 404 if not found                           |
| POST   | `/api/projects/{id}/bookmark`  | Toggle bookmark. Returns `{ "bookmarked": true }` or `{ "bookmarked": false }`. Returns HTTP 404 if not found |

## What to Implement

### Backend — `backend/routers/projects.py`

Implement the three endpoint functions. Each has a docstring describing the expected behavior.

Requirements:
- Use SQLAlchemy (no raw SQL strings)
- At least one endpoint must use `async def`
- Return proper HTTP status codes (200, 404)

### Frontend — `frontend/src/`

**`hooks/useProjects.js`**
- Fetches the project list from `GET /api/projects`
- Accepts an optional state filter and re-fetches when it changes
- Exposes: `{ projects, loading, error, filter, setFilter }`

**`components/ProjectList.jsx`**
- Displays all projects in a table or card layout
- Shows per project: name, state, capacity (MW), fuel type, status, and a bookmark indicator
- Includes a dropdown to filter by state (uses the hook's `setFilter`)
- Shows a loading state while fetching and an error message if the fetch fails
- On project click → shows the detail view for that project

**`components/ProjectDetail.jsx`**
- Displays all fields of the selected project
- Renders nothing (or a placeholder message) when no project is selected
- Includes the `BookmarkButton`

**`components/BookmarkButton.jsx`**
- Calls `POST /api/projects/{id}/bookmark` on click
- Updates the bookmark indicator in the UI **without reloading the page**
